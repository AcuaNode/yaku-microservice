# AGENTS.md — equipment-service

## Build & Verify
- `./mvnw clean install` — builds, runs all tests, and produces the JAR.
- `./mvnw test` — runs the full test suite.
- `./mvnw spring-boot:run` — starts the app locally. Requires PostgreSQL running (see below).

## Runtime Dependencies
- **PostgreSQL** is required for local development.
  - `docker compose up -d` spins up `postgres:latest` on port `5432` with DB `yaku_equipment`.
  - `compose.yaml` lives at repo root and includes **Kafka in KRaft mode** (no Zookeeper) on port `9092`.
- **Kafka** is configured for event publishing. The bootstrap server is `localhost:9092` (configured in `application-dev.properties`).
- There is no MQTT or Spring Integration configuration in this service.

## Architecture
- **Spring Boot 3.4.3** microservice extracted from the former YakuControl monolith.
- Only two top-level packages remain under `io.github.rafaviv.yakubackend`:
  - `equipment` — the bounded context (Farm, Pond, Equipment aggregates).
  - `shared` — common base aggregates, OpenAPI config, exception handling, naming strategy.
- Internal package convention per module (hexagonal-ish):
  - `domain.model.*` — aggregates, value objects, commands, queries, events
  - `domain.services.*` — service interfaces (ports)
  - `application.internal.*` — service implementations and event handlers (adapters)
  - `infrastructure.*` — JPA repos, external adapters, seeders, config
  - `interfaces.*` — REST controllers, resources, assemblers, ACL facades
- Cross-service communication is async via **Kafka** topics (`pond-events`, `equipment-events`). There are no in-memory Spring domain events.
- `shared.domain.model.aggregates.AuditableAbstractAggregateRoot` is the base class for aggregates; JPA auditing is enabled.
- Table naming uses a custom `SnakeCaseWithPluralizedTablePhysicalNamingStrategy`.

## Testing
- **Unit tests** are plain JUnit 5; domain aggregate tests live in `src/test/java/.../domain/model/aggregates/`.
- **Service layer unit tests** cover ownership validation: `FarmCommandServiceImplTest` and `PondCommandServiceImplTest` under `application/internal/commandservices/`.
- **Integration tests** override the datasource to an **H2** in-memory DB and disable SQL initialization (`spring.sql.init.mode=never`). If you add new Spring Boot tests that load the full context, copy the `@TestPropertySource` pattern from existing integration tests.
- **No BDD/Cucumber tests** — feature files were removed during extraction (inactive and out-of-scope).

## Security & API
- **No JWT or Spring Security** inside this service. Authentication is handled upstream by the API Gateway.
- User identity is injected via the `X-User-Id` HTTP header. Controllers accept it with `@RequestHeader("X-User-Id") Long ownerId`.
- **Equipment ownership**: `EquipmentController` does **not** receive `X-User-Id`. Equipment is scoped by `pondId`/`farmId`. The API Gateway must always include `farmId` or `pondId` query params when calling `GET /api/v1/equipment`. Without query params, this endpoint returns **all equipment** (no owner filtering).
- OpenAPI/Swagger UI is available at `/swagger-ui/index.html` when the app is running.
- API base path for controllers is `/api/v1/*`.

## Gotchas
- When running behind the API Gateway, start the service on port **8081** (`./mvnw spring-boot:run -Dspring-boot.run.arguments="--server.port=8081"`).
- `SnakeCaseWithPluralizedTablePhysicalNamingStrategy` lives in this service's `shared` package. Other services (e.g. `notification-service`) may reference it; copying or removing that reference is their responsibility.
- `data.sql` only seeds equipment-relevant data (no IAM roles, users, or telemetry samples). It runs on startup when using the default profile.
- `api-docs.json` at repo root is a static OpenAPI spec snapshot; editing it does not change the runtime API. The live spec is generated by `springdoc`.
- When running locally, use the `dev` profile (`--spring.profiles.active=dev`) or ensure `application-dev.properties` is present so the service connects to `yaku_equipment` instead of the old monolith database.
